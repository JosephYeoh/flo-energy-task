Q1. What is the rationale for the technologies you have decided to use?
- `TypeScript` + `Node.js` for strong typing, safer refactors, fast iteration, and excellent streaming I/O support for large files.
- `csv-parse` for robust CSV handling suited to NEM12 data and is well maintained.
- Node streams + `pipeline` for deliberate backpressure handling and safe large file I/O.
- Docker for consistent runtime and easy deployment.

Q2. What would you have done differently if you had more time?
- Integrate with a message-per-file queue worker (e.g. Kafka) with visibility timeouts and retry policies.
- Implement full 400/500 record support (quality flags, reason codes, B2B metadata).
- Add a file format checker to validate files before processing.
- Improve observability with metrics (rows/sec, files/sec), error rates, and structured logging.

Q3. What is the rationale for the design choices that you have made?
- Coding choices:
    - Streaming processing keeps memory bounded for very large inputs.
    - Batched SQL generation reduces DB overhead and improves ingestion throughput.
    - Batch INSERTs with ON CONFLICT upsert for idempotency and lower DB overhead.
    - Output uses `pipeline` to respect backpressure when writing large SQL files.
    - Precompute interval offsets to reduce per-row computation.

- Specifications choices:
    - Use period-ending timestamps per NEM12 spec; table uses timestamp without timezone.
    - Skip and log missing/invalid consumption values because real-world inputs can be messy, while keeping an audit trail.
    - Fail on malformed structural records to avoid silent corruption.

- Operational choices:
    - Structured logs (using --logs-dir flag) for execution summary and any errors/skipped lines.
    - The `processByFileName()` abstraction keeps per-file execution isolated, making it straightforward to wire into Kafka or any message-per-file queue.
    - Per-file failures are isolated so one bad input doesn’t block the rest.
